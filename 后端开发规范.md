# Python 后端开发规范与脚手架指南（FastAPI版）

## 0. 框架选择：FastAPI

本项目后端选用 **FastAPI** 框架。

- **原因**：原生支持类型提示，IDE 补全友好，降低出错率；自动生成交互式 API 文档（Swagger UI），方便与 Web 和微信小程序前端联调；异步支持好，性能优越。

## 1. 环境与依赖规范

### 1.1 Python 版本

- 强制使用 **Python 3.10** 及以上版本，以充分利用类型提示和模式匹配。

### 1.2 依赖管理

- 推荐使用 **Poetry** 或 **uv** 进行依赖管理，抛弃传统的 `requirements.txt`。
- 必须提交 `poetry.lock` 或 `uv.lock` 文件到版本控制，确保所有开发者和生产环境的依赖版本绝对一致。

## 2. 标准脚手架目录结构

良好的目录结构是项目成功的一半，本项目采用标准的分层架构（Controller - Service - Model）：

```
project_root/
├── app/
│   ├── __init__.py
│   ├── main.py                 # FastAPI 实例初始化与程序入口
│   ├── api/                    # 路由层 (Controller)
│   │   ├── dependencies.py     # FastAPI 依赖注入（如获取当前用户、数据库会话）
│   │   ├── endpoints/          # 具体的 API 路由（如 user.py, order.py）
│   │   └── routers.py          # 路由汇总注册
│   ├── core/                   # 核心配置
│   │   ├── config.py           # 环境变量与全局配置 (pydantic-settings)
│   │   ├── security.py         # JWT 签名、密码哈希等安全逻辑
│   │   └── exceptions.py       # 自定义异常类与全局异常处理器
│   ├── models/                 # 数据库模型层 (SQLAlchemy ORM)
│   │   ├── base.py             # declarative_base
│   │   └── user.py             # 具体的数据库表映射
│   ├── schemas/                # 数据校验与序列化 (Pydantic Models)
│   │   ├── base.py             # 统一响应体模型
│   │   └── user.py             # 请求入参 (UserCreate) 和出参 (UserResponse) 结构
│   ├── services/               # 业务逻辑层 (Service)
│   │   └── user_service.py     # 核心业务逻辑，供 api 层调用
│   └── utils/                  # 工具函数与第三方 SDK
│       └── wechat.py           # 微信小程序鉴权、支付等接口封装
├── tests/                      # 测试目录
│   ├── conftest.py             # pytest 全局配置与 fixture
│   └── api/                    # 接口自动化测试用例
├── alembic/                    # 数据库迁移脚本目录 (Alembic)
├── .env                        # 本地环境变量配置（不提交到Git）
├── .gitignore                  # Git 忽略文件配置
├── pyproject.toml              # 项目依赖与工具配置
└── Dockerfile                  # 容器化构建脚本
```

## 3. 编码与 Git 规范

### 3.1 代码风格

- **格式化工具**：强制使用 **Ruff** 或 **Black** 进行代码格式化。配置在 `pyproject.toml` 中，建议开启 IDE 的 "Format on Save"（保存时自动格式化）。

- **类型注解**：所有函数的参数和返回值**必须**包含 Type Hints。

  ```
  # 规范示例
  def get_user_by_id(user_id: int) -> UserResponse | None:
      pass
  ```

- **命名规范**：

  - 变量/函数/模块：`snake_case` (如 `user_id`, `get_user()`)
  - 类名：`CamelCase` (如 `UserModel`, `WechatService`)
  - 常量：`UPPER_SNAKE_CASE` (如 `MAX_PAGE_SIZE = 100`)

### 3.2 提交规范

遵循 Angular 提交规范：

- `feat:` 新功能 (Feature)
- `fix:` 修复 Bug
- `docs:` 文档变更
- `refactor:` 代码重构（不增加新功能，也不修复bug）
- `test:` 增加测试用例

## 4. API 接口设计规范 (针对 Web & 微信小程序)

### 4.1 统一响应格式

所有 API 请求必须返回一致的 JSON 结构。在 FastAPI 中可通过自定义 Response 类或统一封装实现：

```
{
    "code": 20000,         // 业务状态码：20000表示成功，非20000表示业务错误
    "message": "登录成功", // 提示信息，可直接展示给用户
    "data": {              // 实际有效数据
        "token": "eyJhbG..."
    }
}
```

### 4.2 多端登录鉴权逻辑

系统采用 **JWT (JSON Web Token)** 进行无状态鉴权。

- **Web 端登录**：前端提交账号/密码，后端验证通过后生成 JWT 返回，前端将其放入 HTTP 请求头的 `Authorization: Bearer <token>` 中。
- **微信小程序登录**：
  1. 小程序端调用 `wx.login()` 获取 `code`。
  2. 小程序将 `code` 发送给后端 API。
  3. 后端 `utils/wechat.py` 携带 `AppID`, `AppSecret`, `code` 请求微信服务器，换取 `openid` (和 `unionid`)。
  4. 后端通过 `openid` 查找用户。若不存在则自动注册（或要求绑定手机号）。
  5. 登录/注册成功后，后端生成基于内部 `user_id` 的 JWT 返回给小程序。
- **业务处理一致性**：无论是 Web 还是小程序，`api/dependencies.py` 中的 `get_current_user` 依赖项只负责解析 JWT 并提取 `user_id`，底层业务逻辑（Services）对前端来源无感知。

### 4.3 RESTful 规范

- 使用 HTTP 方法表达动作：`GET` (查询), `POST` (创建), `PUT` (全量更新), `PATCH` (局部更新), `DELETE` (删除)。
- URL 使用名词复数，如 `/api/v1/users`，不使用动词如 `/api/v1/get_users`。

## 5. 异常与日志规范

### 5.1 全局异常处理

- 严禁在路由层写 `try...except Exception` 来捕获一切。
- 在 `core/exceptions.py` 定义自定义业务异常类（如 `UserNotFoundError`）。
- 在 `main.py` 中注册 FastAPI 的 `@app.exception_handler()`，统一将业务异常转化为标准的 JSON 响应格式（见 4.1）。

### 5.2 日志记录

- 使用 `Loguru` 替代标准库 `logging`，配置更简单。
- **INFO**：记录关键业务流转（如“用户1024下单成功”）。
- **ERROR**：记录代码异常、第三方API调用失败等，需对接警报系统。

## 6. 测试与联调规范

### 6.1 接口联调

- 利用 FastAPI 自动生成的文档（`http://localhost:8000/docs`）进行调试。
- 后端开发应保证文档中的请求示例（Example Values）准确无误，前端可直接根据 Swagger 文档生成请求代码。

### 6.2 自动化测试

- 使用 **pytest** 框架。
- 核心业务层（`services`）必须编写单元测试。
- 对接微信接口等第三方服务时，在测试中使用 `unittest.mock.patch` 模拟返回值，禁止测试用例直接发起真实的外部网络请求。

## 7. 部署与 CI/CD 规范

### 7.1 运行方式

- **开发环境**：`uvicorn app.main:app --reload`

- **生产环境**：强烈建议使用 `Gunicorn` 管理 `Uvicorn` 工作进程，以充分利用多核 CPU。

  命令示例：`gunicorn app.main:app -w 4 -k uvicorn.workers.UvicornWorker`

### 7.2 容器化 (Docker)

项目必须包含 `Dockerfile`。使用多阶段构建或基于 `python:3.10-slim` 减小镜像体积。敏感配置（如数据库密码、微信 AppSecret）必须通过 Docker 环境变量传入，绝不能写死在代码或镜像中。

### 7.3 CI/CD 流水线

- **CI (持续集成)**：在代码提交到 `main` 或 `develop` 分支时，自动触发代码格式检查（Ruff）和单元测试（pytest）。
- **CD (持续部署)**：测试通过后，自动打包 Docker 镜像并推送到镜像仓库，随后通知服务器触发容器更新。
